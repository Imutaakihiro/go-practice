## 2026-04-20 13:11 学習スタンス：「なんとなく書けて読める」を最初のゴールに

学び始めの段階では、**完璧に理解してから次へ** ではなく **なんとなく書けて読める** くらいで先に進むことが大切。

- Goは文法が小さく、パターンマッチで読める範囲が広いので、特にこの戦略が効く

## 2026-04-20 13:22 はじめてのエラー：`go run` のパス指定

Loop 1 で `01.go` を初めて実行しようとして、エラーになった。一緒に読み解いた記録。

### 起きたこと

`learning/basics/examples/` ディレクトリにいる状態で:

```
go run ./learning/basics/examples/01.go
```

を実行 → `stat ./learning/basics/examples/01.go: no such file or directory`

### エラーメッセージの分解

```
stat ./learning/basics/examples/01.go: no such file or directory
└─┬─┘ └────────────────────────────┘  └─────────────────────────┘
  │              │                              │
  │              │                              └─ 「そのファイルもディレクトリも無い」
  │              └─ Goが探しに行ったパス（現在地起点で解釈される）
  └─ stat = ファイル情報を取りに行くシステムコール。これが失敗した
```

### 原因

- 既に `examples/` の中にいるのに `./learning/basics/examples/01.go` と書いた
- `./` は **「今いる場所」** を意味する → Goは `examples/learning/basics/examples/01.go` を探しに行った
- そんなパスは存在しない → `no such file or directory`

### 正しい指定の仕方（2パターン）

| 起点 | コマンド | いつ使う |
|---|---|---|
| 今いる場所が `examples/` 内 | `go run ./01.go` または `go run 01.go` | サクッと試したいとき |
| リポジトリルート | `go run ./learning/basics/examples/01.go` | CLAUDE.md推奨。標準的な書き方 |

### 学んだこと

- `./` は **絶対パスの記号ではなく、相対パスの起点（=現在地）** を表す
- パスの先頭に `./` を付けるかどうかで意味が変わる:
  - `./01.go` → 「現在地にある 01.go」
  - `01.go` → 同じ意味（多くの場合）
  - `/Users/.../01.go` → 絶対パス（どこから打っても同じファイルを指す）
- エラーメッセージの **`stat ...: no such file...`** を見たら「**Goが探したパスを書き出してくれている**」と覚える。そこに自分の現在地起点で解釈された結果が出ているので、`pwd` と見比べると原因がすぐわかる

### 次から気をつけること

- コマンドを打つ前に `pwd` で **今いる場所を確認** する癖をつける
- 迷ったら **リポジトリルートに戻ってから** `go run ./learning/...` で打つ（CLAUDE.md推奨スタイル）

## 2026-04-20 13:53 はじめての syntax error：`:=` は関数の中だけ

変数宣言の練習中、`age := 30` を `func main()` の **外側** に書いて実行 → エラーになった。

### 起きたこと

```go
package main

import (
	"fmt"
)

age := 30          ← ここ（func の外側）

func main() {
	fmt.Println(age)
}
```

実行したら:

```
learning/basics/examples/01.go:7:1: syntax error: non-declaration statement outside function body
```

### エラーメッセージの分解

```
learning/basics/examples/01.go : 7 : 1 : syntax error : non-declaration statement outside function body
└─────────────┬─────────────┘ └┬┘ └┬┘ └─────┬─────┘ └──────────────────┬──────────────────────┘
       問題のファイル           行  列   書き方の違反         「やることリストの外に
                              番  番                          宣言じゃない命令文があるよ」
```

### 原因

- `:=` は **`func 〇〇() { ... }` の中でしか使えない** ルール
- 関数の外で値に名前をつけたいなら `var age = 30` を使う
- ご主人様は `:=` をいきなり外で使ったので、Goが「ここに置いていい命令じゃないよ」と怒った

### 直し方（採用したやつ）

`age := 30` を `func main() { ... }` の中に移動した:

```go
func main() {
	age := 30
	fmt.Println(age)
}
```

→ 画面に `30` と出て成功。

### 学んだこと

| 書き方 | 書ける場所 | 型を書く必要 |
|---|---|---|
| `age := 30`（短縮形） | `func ○○() { ... }` の中だけ | 不要（Goが推測） |
| `var age int = 30`（正式） | どこでも（中・外） | 必要 |
| `var age = 30`（型省略） | どこでも（中・外） | 不要（Goが推測） |

**今後の基本方針**: 普段は **`:=` を使う**。`:=` が使えないシーン（関数の外で書きたいとき、まず宣言だけして後で値を入れたいときなど）でだけ `var` を出す。

## 2026-04-20 13:53 おまけ：型推論ってGoだけの話じゃない

`:=` で型を書かなくていいのは **Goが値から型を推測してくれる** から。これを **型推論** と呼ぶ。

```go
age := 30        ← Go「30 は整数 → age は int 型」
name := "ご主人様" ← Go「"..." は文字列 → name は string 型」
pi := 3.14       ← Go「3.14 は小数 → pi は float64 型」
flag := true     ← Go「true は真偽値 → flag は bool 型」
```

### 他言語にもある

型推論は最近の言語のトレンドで、Goだけのものじゃない。**TypeScript / Rust / Swift / Kotlin / C#** などにも入っている。

昔の言語（古いCやJavaなど）は型を必ず人間が書くルールだったが、「**見て分かることを機械にやらせよう**」という流れで、新しい言語は型推論を標準装備するようになった。

### エンジニアの基本思想として

「**人間がいちいち書かなくていいことは機械にやらせる**」という考え方は、Goに限らずソフトウェア開発全般の基本思想。これに気づけたのは大きい。

## 2026-04-20 14:02 なぜ「外部公開」と「内部限定」を分けるのか（カプセル化の話）

「先頭の大文字／小文字で公開／非公開が分かれる」と聞いて、「**別にどっちでもよくない？**」と疑問に思ったので深掘り。

### 一言でいうと

**「外に見せたものだけが "約束"。それ以外（内部）は、後で自由に作り直せる」** ようにするための設計。

### ラーメン屋の例えで理解

- **お客に見せる（公開）**: メニュー、注文方法
- **厨房（非公開）**: スープの配合、麺のゆで時間

もしお客が厨房に入って勝手にスープを変え始めたら大混乱する → だから外向けと内部を分ける。

### プログラムでも同じ

```
公開する道具:
  - CalcTaxIncluded(価格)   ← 大文字始まり = 外から呼べる

内部で使う道具（隠す）:
  - getCurrentTaxRate()    ← 小文字始まり = 同じパッケージ内だけ
  - roundFractions(値)
```

- 外から見えるのは `CalcTaxIncluded` だけ → シンプル
- 内部の作り方は **後で自由に変えられる**（外に影響しない）

もし全部公開していたら → 他の人が内部関数も使い始める → **変更できない呪いにかかる**

### なぜ「エンジニアの基礎」なのか

ソフトウェアは **何年も改修し続けるもの** 。

- 内部は **どんどん良くしていける**
- 外向けの約束だけ守っていれば、誰にも迷惑をかけない

これが **カプセル化（capsulation）** という考え方。カプセル薬のように、中身を包んで見せない設計。

### 1人で書いてる学習用でも意味があるか？

ある。理由:

- **3ヶ月後の自分は他人**。「公開されてる名前 = ここが入口」とすぐ分かる
- 仕事でチーム開発するときに、いきなり身につくものではない。学習中から染み込ませておくのが得

### なぜGoは「名前の大文字／小文字」で表現する？

他の言語は `public` `private` というキーワードを書くが、Goは「**キーワードを書かせると面倒。名前の最初の文字だけで表せばシンプルだろう**」という設計哲学。書く文字数を減らす方を選んだ。

### 学んだこと

| 用語 | 意味 |
|---|---|
| **カプセル化** | 中身を包んで、外には窓口だけ見せる設計思想 |
| **公開（public）** | 外から呼べる。Goでは大文字始まり |
| **非公開（private）** | 内部限定。Goでは小文字始まり |

これを意識し始めると「これは外に出していい？それとも内部用？」と自然に考えるようになる → これが **設計力** の入口。

- `println`（小文字）は存在しない or 内部限定 → 外から呼べない

**ルールを学んだ直後に実際エラーで踏むパターン** は最強の定着方法。

### エラー③ `main` の重複：実行コマンドの問題

```
go run ./learning/basics/examples/
                                  ↑ スラッシュで終わるとフォルダ指定
```

エラー: `main redeclared in this block`

#### 原因

`go run ./フォルダ/` と打つと、**そのフォルダ内の Go ファイル全部** をまとめて実行しようとする。`examples/` には `01.go` と `02_introduction.go` の両方に `func main()` があるので、**「同じパッケージに main が2つ」** になって衝突。

#### Goのルール

> **同じパッケージに `main` 関数は1つだけ** （社長は1人ルール）

#### 直し方

ファイル名まで指定する:

```
go run ./learning/basics/examples/02_introduction.go
```

→ `01.go` は無視されて、`02_introduction.go` だけがコンパイルされる → main は1個 → 動く。

### おまけ：`main` を別の名前にすればいいのでは？

「`main` を `introduce` にリネームすれば衝突しないのでは？」と考えたが、**それはNG** 。理由:

- `main` は **Goが探しに行く特別な名前** （変えられない）
- リネームすると `function main is undeclared in the main package` でエラー
- `main` は **「Go が呼ぶ専用の予約名」** なので、人間の都合で変えられない

→ 解決策は2つだけ:
1. **`go run` でファイル名を指定** （現状の運用）
2. 各 main を **別フォルダに分ける** （本格的な構成、学習中は不要）

### 学んだこと

| ポイント | 教訓 |
|---|---|
| エラー数 ≠ 間違い数 | 5個のエラーが出ても、本当の間違いは2種類だけだった |
| Goのエラーは親切 | `(but have Println)` のようにヒントを出してくれる。最後まで読む |
| `go run` の指定先 | ファイルなら1ファイル、フォルダなら中の全部 |
| `main` は特別 | 重複NG、リネームNG。Go の「予約された名前」 |

## 2026-04-20 14:30 おまけ：`func init()` という "もう1つの特別関数"

polarisの本番コードで `func init()` を見つけて疑問に思ったので調査。

### 一言でいうと

> **「`main` より先に、Goが自動で呼ぶ "準備関数"」**

### 実行順序

```
プログラム起動
    ↓
    ├─ init() を全部呼ぶ        ← 開店準備フェーズ（複数あれば順番に）
    ↓
    └─ main() を1回呼ぶ         ← 本番フェーズ
```

### コード例

```go
func init() {
    fmt.Println("【起動準備】設定読み込み完了")
}

func main() {
    fmt.Println("【メイン処理】開始")
}
```

実行結果:
```
【起動準備】設定読み込み完了
【メイン処理】開始
```

### `init` の特徴

| 特徴 | 内容 |
|---|---|
| 呼び方 | **Goが自動で呼ぶ**（人間は呼ばない） |
| 順番 | **`main` より先**に必ず走る |
| 引数・戻り値 | 持てない |
| 数 | 同じパッケージに複数書いてOK |
| 名前 | **小文字 `init` 固定** （`Init` はNG） |

### 何のために使うか（本番でよく見る用途）

1. **設定の準備** — 環境変数や設定ファイルを読む
2. **ライブラリへの登録** — `import _ "github.com/lib/pq"` の `_` パターンで、ライブラリ側の init が勝手にドライバを登録
3. **起動チェック** — 必須の環境変数がなければ `main` に入る前にエラーで止める

### `main` と `init` は「家族」

両方とも **Goが自動で呼ぶ "裏方の名前"** 。だから:
- 名前は **固定**（人間が変えられない）
- **小文字**（外向けに公開する関数じゃない）

「**Goが内部で呼ぶ予約名**」というカテゴリで、`main` と `init` は同じ家族。

### 自分で書くべき？

**学習段階では書く必要なし** 。本番コードで見かけたら「**あ、起動準備してるんだな**」と読めれば十分。

## 2026-04-20 Loop 2 — 型と変換 + ゼロ値

### 学んだ3つの概念

1. **型 = 入れ物の形**
   - `string`（文字）/ `int`（整数）/ `float64`（小数）/ `bool`（はい/いいえ）
   - 一度決まった型は変えられない（`age := 30` したら、後から `age = "三十"` はできない）

2. **型変換は明示的にやる**
   - `float64(age)` のように「変換してください」と書く必要がある
   - Goは勝手にやってくれない（暗黙の型変換禁止）

3. **ゼロ値**
   - 変数を宣言だけしても、自動で初期値が入る（`int` → 0, `string` → ""）
   - 「不確実な状態を作らない」哲学の現れ

### 書いたコード

`examples/03_bmi.go` で BMI計算プログラムを作った。
- 身長 `float64` と体重 `int` を混ぜて計算する場面で、型変換 `float64(weight)` が必要
- `bmi := float64(weight) / (height * height)` で計算成功（22.49...）

### わざとエラーを出してみた

**実験A**: `bmi := weight / (height * height)` (型変換なし) で実行
```
invalid operation: weight / (height * height) (mismatched types int and float64)
```
- `mismatched types` = 「型が一致してない」のサイン
- コンパイル時に止まってくれる = 動かす前にバグが見つかる安心感

**実験B（予想だけ）**: `var height int = 1.7` は「1として認識される」と予想したが、実際は…
```
cannot use 1.7 (untyped float constant) as int value in variable declaration (truncated)
```
- Goは **「1にしてあげる」ことすらしない**
- データが失われる変換は自動でやらない（責任を取らせる設計）
- `int(1.7)` と明示的に書けば `1` になる（自分で「切り捨ててOK」と宣言する形）

### 気づき・哲学

- 多くの言語は暗黙の型変換をしてくれる（楽）。Goはしない（明示性と安全を優先）
- 最初は面倒に感じるけど、デカいプログラムになった時に「気づかないうちに精度が落ちてた」事故を防げる
- エラーメッセージは `file:line:col: 種類: 内容` の構造。慣れると30秒で原因がわかる
- `untyped float constant` という用語が出てきた → リテラルは最初「型なし」で、代入先で型が決まる仕組みらしい（深掘りはまた今度）

## 2026-04-20 補足 — 「他の言語は型変換どうしてるの?」

Loop 2 のあと、ふと疑問に思って聞いてみた: 「他の言語だったら `int x = 1.7` って書いたら勝手に 1 にしてくれるよね?」

→ 答え: **言語によって3パターンに分かれる**

| 振る舞い | やる言語 | `int x = 1.7` の結果 |
|---|---|---|
| 黙って切り捨てる（楽だけど危険） | C, C++ | `x = 1`（警告は出ることも） |
| 自動で「ゆるい型」に揃える | Python, JavaScript | エラーにせず `1.7` のまま保持 |
| 拒否する（明示しろと怒る） | **Go**, Java, Rust | コンパイルエラー |

### 気づき

- 自分の予想「1になる」は **C系の言語の挙動** だった。古い言語は「速さ・楽さ」優先
- 「黙って切り捨て」が原因のバグが世界中で発生しまくった反省で、新しい言語（Go, Rust）は「危険な変換は明示的に書け」設計になってる
- 言語の設計は **「過去の事故から学んだ反省」** の積み重ね
- 「楽 vs 安全」のトレードオフ。スクリプト用途は楽な方が便利、大規模システムは安全な方が向いてる
- だから **「言語に優劣はない」、用途で選ぶ** という感覚を持つと、新しい言語を学ぶときも入りやすい

## 2026-04-20 補足 — 関数(func)の基本構文（Loop 3で先取り）

ポインターの説明で `func birthday(age int) { ... }` が出てきて「これ何?」と聞いた → 最小限を先取りで教えてもらった（深入りは Loop 6 でやる予定）。

### 4つのパーツ

```
func   birthday   (age int)   { ... }
 ↑        ↑          ↑          ↑
 ①        ②          ③          ④
```

| パーツ | 意味 |
|---|---|
| ① `func` | 「関数ですよ」の合図（決まり文句） |
| ② `birthday` | 関数の名前（自分で決める） |
| ③ `(age int)` | 受け取る材料（引数=ひきすう） |
| ④ `{ ... }` | 関数の中身（やる作業） |

### 引数 `(age int)` の構造

`age` = 名前 / `int` = 型 → 「`int` 型のものを `age` という名前で受け取りますよ」の宣言。

### 日常の例え: レシピ

- レシピ名 = 関数名（`birthday`）
- 「材料: 卵2個、砂糖50g」 = 引数（`age int`）
- 「手順: ①卵を割って... ②砂糖を混ぜて...」 = 中身（`{ ... }`）

レシピは「材料を渡されたら、手順通りに作って完成品を返す」仕組み。関数も全く同じ。

### 呼び出し

```go
birthday(30)  // 30 が関数の中の age として受け取られる
```

### ポインター版

```go
func birthday(agePtr *int) {
    //         ↑      ↑
    //         名前    型（intの住所）
}
```

型のところを `*int` にすると **「intの住所」を受け取る** という意味になる。

### 気づき

- `func main()` も同じ形。引数なし(`()`が空) の関数
- 関数 = 「処理に名前をつけて、何度でも呼び出せるようにする」仕組み = プログラムの部品化
- Loop 6 で **戻り値**（結果を返す仕組み）と **複数戻り値**（Goの特徴）を本格的にやる予定

## 2026-04-20 Loop 3補足 — `&` と `*` の方向、ポインター変数の命名

ポインターを学んでて2つ詰まった→腑に落ちた話。

### 詰まり①: 「`agePtr := *なにか` って書けないの?」

`*` で中身が取れるなら、`*` でポインター作れそうな気がしたけど → **書けない** 。

**理由**: `&` と `*` は **逆向きの操作** だから。

```
[家(変数)] ──(&)──> [住所(ポインター)] ──(*)──> [家(変数の中身)]
                住所を取る             住所をたどる
```

| 出発点 | 操作 | 結果 |
|---|---|---|
| 変数 `age` | `&age` | 住所（ポインター） |
| 住所 `agePtr` | `*agePtr` | 中身（値） |

ポインター（住所）を作るには **必ず `&`** が必要。`*` では住所は作れない（中身しか取れない）。

### 4パターン整理

| 書き方 | 意味 | 例 |
|---|---|---|
| `age` | 中身そのもの | `30` |
| `&age` | ageの住所 | `0xc000012028` |
| `agePtr` | 中身は住所 | `0xc000012028` |
| `*agePtr` | 住所をたどった中身 | `30` |

→ `age` と `*agePtr` が同じものを指す。`&age` と `agePtr` が同じものを指す。

**覚え方**: 
- `&` をつけると一段「外」に出る
- `*` をつけると一段「中」に入る
- 階段の昇り降りのイメージ

### 詰まり②: 「`agePtr` って名前じゃないとダメ?」

→ **何でもいい！** `banana := &age` でも動く。`*banana` で中身取れる。

`Ptr` は **「これはポインターですよ」と読み手に伝える慣習的な目印** 。`Pointer` の略。

### 命名についての気づき

- コードは **書く時間より読まれる時間の方が長い** → 「ぱっと見で意味がわかる名前」をつける
- Goは **短い名前を好む** 文化（短いスコープなら `p := &age` で十分）
- 名前で「型が何か」を伝える文化（ハンガリアン記法）は古め。今は型システムが守ってくれる
- **「命名は技術であって、ルールではない」** = 読み手のためにつける、が本質

## 2026-04-20 Loop 3 まとめ — ポインター

### 学んだこと

1. **変数には住所がある** — `&age` で取り出せる
2. **ポインター型 `*int`** — 「intの住所を入れる入れ物」
3. **`*agePtr` で住所をたどる** — 中身の取り出し/書き換え両方できる
4. **`nil`** — ポインターのゼロ値、辿るとpanic（強制終了）

### 一番の使い道

**「関数で値を変える」** — Goは基本コピー渡し。元を変えたい時だけ住所を渡す（`&` 明示）。

```go
func depositPtr(balancePtr *int, amount int) {
    *balancePtr = *balancePtr + amount
}
balance := 1000
depositPtr(&balance, 500)  // → balance が 1500 に変わる
```

### 書いたコード

`examples/04_savings.go` で「貯金箱に入金」プログラム。
- ポインターなし版 → 変わらない（コピーが書き換わるだけ）
- ポインター版 → 元が変わる

### nil pointer dereference 体験

`var emptyPtr *int; fmt.Println(*emptyPtr)` で意図的にクラッシュ。
- `panic: runtime error: invalid memory address or nil pointer dereference`
- これは **ランタイムエラー** （実行時に発覚）。コンパイル時には防げない
- 実コードでは `if ptr != nil { ... }` でチェックする → Goの「if err != nil 文化」の下地

### 哲学的な気づき

- Pythonなどは暗黙にポインター渡し（リスト・辞書）→ いつ元が変わるか型から見えない
- Goは「変えたい時だけ明示」→ コードを読むだけで意図がわかる
- 「楽 vs 明示性」のトレードオフ。Goは明示性側
- ガベージコレクションがあるので、C/C++のようなメモリ事故はほぼ起きない

### 詰まったところ → 解消

- 「`agePtr := *なにか` って書ける?」 → 書けない。`&` と `*` は逆向きの操作
- 「`agePtr` って名前じゃないとダメ?」 → 何でもいい。`Ptr` は読み手への目印（慣習）

## 2026-04-20 Loop 4 まとめ — スライス + マップ

### 学んだこと

**「複数のデータをまとめて扱う」道具が2種類ある**

| 道具 | 何が得意 | 引き方 |
|---|---|---|
| スライス `[]string` | 順番に並んだデータ | 番号 `s[0]` |
| マップ `map[string]int` | キーで引きたいペア | キー `m["a"]` |

### 配列とスライスの違い

| | 配列 (`[3]string`) | スライス (`[]string`) |
|---|---|---|
| サイズ | 固定（型に数字が入る） | 可変（appendで追加可能） |
| 使用頻度 | ほぼ使わない | 9割こっち |

→ 実コードでは **スライス** をほぼ使う。配列はスライスの裏側として知っておくだけでOK。

### 文法対比

| やりたいこと | スライス | マップ |
|---|---|---|
| 取り出す | `s[0]` | `m["a"]` |
| 書き換え | `s[0] = "X"` | `m["a"] = "X"` |
| 追加 | `s = append(s, "C")` | `m["c"] = "C"` |
| 削除 | （複雑） | `delete(m, "a")` |
| 要素数 | `len(s)` | `len(m)` |

**マップは「追加」と「書き換え」が同じ書き方** — キーがあるかないかで自動判別。
**スライスの追加だけ特殊** — `append` の戻り値を再代入する必要あり（理由は後で深掘り）。

### 書いたコード

`examples/05_inventory.go` で在庫管理プログラム。
- `products []string` で商品名一覧
- `stock map[string]int` で商品名→在庫数
- `append` でスライス拡張、`stock["ぶどう"] = 3` でマップに追加

### スライスとマップの使い分け

| シーン | どっち |
|---|---|
| 順番が大事（出席番号順） | スライス |
| キーで引きたい（名前→電話番号） | マップ |
| 同じものが複数あってOK | スライス |
| 一意の鍵で管理したい | マップ |

### 大事な気づき

- **「最初は読めればOK、書くのは後で」** — 文法を全部覚える必要なし。プロも毎回ググる
- **意図を読み取る力** が本体。「append してるな」「mapに追加してるな」とわかれば十分
- マップに「ない鍵」でアクセスすると **ゼロ値** が返る（エラーにならない）→ `value, ok := m["key"]` で「ある/ない」を区別できる
- マップは内部的にハッシュテーブルで作られているので、要素数が多くても引くのが速い
- マップの **取り出し順は保証されない** → ループで回すと毎回順番が変わる（後で詰まる人多い）

## 2026-04-21 Loop 5 まとめ — 制御構文（if / for / switch）

### 書いたコード

`examples/06_grade.go` で成績表プログラムを作った。名前スライスと点数スライスから、名前・点数・成績・コメントを1人ずつ表示する。

- `gradeOf(score int) string` — if-else if で **点数 → ランク（"A" / "B" / "C" / "D"）** に変換する関数
- `commentOf(rank string) string` — switch で **ランク → コメント文** に変換する関数
- `main` の `for range` — 2つのスライスを番号で束ねて、3つの部品をつなぐ

### 学んだ3つの概念

1. **`for range` は「番号と中身のペア」を返す**
   - `for index, value := range names` の形で、**1つ目が番号、2つ目が中身** という位置だけが決まっている
   - **受け取る名前は自由**（`i, v` でも `banana, apple` でも動く）
   - 番号は **0から始まる**
   - 番号いらないなら `for _, name := range names`、中身いらないなら `for i := range names` で省略可
   - `index` を宣言して使わなかったら `declared and not used` でコンパイルが止まる → 「使わない変数は書き間違いの温床」というGoの哲学（安全側に倒す設計）

2. **index を「JOINキー」として使う発想**
   - `names` と `scores` は同じ番号で対応している → `for index, name := range names { score := scores[index] }` で束ねる
   - 一瞬で **「これINNER JOINだ！」** と BigQuery脳が発動した
   - プログラミングの「同じキーで揃ったデータを束ねる」は **SQLと同じ思想** 。得意領域の感覚がそのまま転用できた

3. **関数の戻り値を `:=` で受け取る**
   - `grade := gradeOf(score)` の形で、関数が返した結果を変数に入れる
   - 関数のシグネチャ `func gradeOf(score int) string` の **最後の `string` が「返す型」の宣言**
   - `return "A"` で呼び出し元に値が返る
   - **自販機の例え**: 小銭を入れる = 引数、商品が出てくる = 戻り値、「お茶が出てきます」の札 = 戻り値の型
   - **バケツリレー**: `score → gradeOf → grade → commentOf → comment` と部品を連続で繋げる

### "アハ" モーメント

- 4〜5行のループが、**1000人いても動く** ことの威力を実感（繰り返しの本質）
- ループ変数を `value` → `name` にリネームすると読み味が変わる。**命名は嘘をつかない** が原則
- 同じ `:=` が「値を入れる」「関数の結果を入れる」両方に使える → **右側が「値」か「関数呼び出し」かだけの違い** で、代入の仕組みとしては同じ

### エラー・詰まり体験

- `index` を宣言して使わなかったらコンパイル拒否 → 「**使わない変数はバグの温床**」というGoの哲学を体感
- 回避策は `_` で受け取って捨てるだが、今回は index を scoresの引き込みに使う方向で解決（`_` は「受け取るけどいらない」のサイン）

### 命名についての気づき

- `range` が返す2つの値の **名前は自由** （位置だけ固定）
- ただし **嘘をつかない・読み手に意味が伝わる名前** を選ぶのが作法
- Goコミュニティの慣習:
  - 短いループなら `i, v`
  - 長くて意味のあるデータなら `index, name` のように説明的に
- 日本語識別子も動くが英語推奨

### 3つの制御構文の使い分け

| 構文 | 得意なこと | 今回の使い方 |
|---|---|---|
| `if / else if` | **範囲の判定**（>=80, >=60 など連続した境界） | `gradeOf` で点数 → ランク |
| `switch` | **値の分岐**（"A" ならこれ、"B" ならこれ、と離散的） | `commentOf` でランク → コメント |
| `for range` | **コレクションを端から端まで** | main で名前と点数を束ねて表示 |

→ 「範囲は if、離散は switch、繰り返しは for」という感覚。ツールごとに **得意な形** がある。

### 裏ゴールへの繋がり

- 「同じ型のデータをコレクションで束ねる」「キーで結合する」「部品を組み合わせる」は **Goに限らずプログラミングの基礎的考え方**
- SQL発想が Go のループ・スライス操作に転用できた = **技術の「自分ごと化」の成功例**
- 得意領域（BigQuery/dbt）で培った「データを揃えて束ねる感覚」はそのまま使える。新しい言語は **ゼロから覚え直す必要はない** と体感できた回

## 2026-04-21 メタ学習 — 「すぐ聞く vs 自分で考える」の判断プロトコル

### 問題意識

学習中に詰まったとき、**AIにすぐ聞くべきか、もう少し粘るべきか** で毎回迷う。これは文法知識より先にある、**学習の質そのものを左右するメタスキル**。ここを雑にすると学習全体の伸びが鈍る。

### 両極はどっちも失敗する

- **考え続ける派**: 時間を溶かしてモメンタムが崩壊する。気づいたら「楽しい」が消えていて、机に向かうのがしんどくなる
- **すぐ聞く派**: 受動学習モードに逆戻り（vibeコーディング期の再来）。AIが答え配達人になって、自分の筋肉が育たない

→ どちらか一方ではなく、**状況に応じて使い分ける判断力** こそが「本当の学習力」。

### 判断軸: 5つのセルフチェック質問

詰まったときに、聞く前にこの5つを自分に投げる。

1. **「何がわからないか」自体を言語化できてる?** → No なら即聞いてOK（情報不足のまま考えても時間が溶けるだけ）
2. **エラーメッセージやヒント、ちゃんと読んだ?** → 読まずに聞くのはダメ。情報補給の前にまず手持ちの資料を使い切る
3. **手持ちの武器で解けそうな問題?** → 解けそうなら5〜10分粘ってみる（ここが筋トレタイム）
4. **詰まってから何分経った?** → 15分以上、糸口すら見えないなら聞く
5. **モメンタムはまだ元気?** → 疲れてきたら素直に聞いて前に進める方が学習効率がいい

### ベストな質問の形 = **仮説ベース**

> 「XXXだと思うんだけど、YYYで合ってる?」

この形だと何が起きるか:

- 自分の **仮説がある** 状態で質問している
- 確認したい点が **ピンポイントに絞れてる**
- 答えが返ってきた瞬間、「自分の仮説の **どこが正しくて、どこがズレてたか**」が整理される
- つまり「聞く」行為そのものが **自分の理解の答え合わせ** になる

NGな聞き方:

- 「どう書けばいい?」 → 仮説ゼロ、丸投げ
- 「エラー出ました」 → 何を試したか、どこまで読んだかが伝わらない

### 自分用プロトコル（合意版）

| ルール | 内容 |
|---|---|
| 5分ルール | まず5分は自分で考える・自分で読む |
| 15分ルール | 糸口も見えないまま15分経ったら聞く |
| エラーは必ず自分で読む | 先にAIに訳させない。自分がわかったところまで伝えてから聞く |
| 答えをもらったら自分の手で書き直す | コピペ式の受動学習を避ける。手を動かして初めて筋肉になる |

### 裏ゴールとの繋がり

「自分で書ける・自分で直せる」状態になるには、**「わからないまま粘る筋力」** が必要。でもそこで折れたら意味がない。だから:

- **筋トレ（粘る）** と **情報補給（聞く）** を **時間でスパッと切る**（感覚で決めない）
- 質問の質を上げる（仮説ベース）ことで、**「聞く」も学習の一部** に変わる

vibeコーディング期からの脱却は **「AIを答え製造機にしない」** が鍵。AIは **学びのパートナー**であって、答え配達人じゃない。この区別がつくようになってきたのが今日の一番大きい収穫。

### 今日の実例: ベストな聞き方だった瞬間

`07_stats.go` を書いている最中の「**nums[0] はゼロ値ですか!**」という質問は、まさに仮説ベースの形になっていた:

- 自分の仮説がある → 「`0` と `0` がついてるから、ゼロ値と何か関係があるのでは?」
- 確認したい点がピンポイント → Yes/No で答えられる粒度
- 学びが残った → Yes + 補足だけで、自分の中で一気に整理がついた

→ 今後は意識的に、この **「XXXだと思うけど、YYYで合ってる?」** の形で聞く癖をつける。

### 気づき

- **「質問できること」も技術の一部**。いい質問は、半分答えが出てる状態
- エンジニア人生は「わからないことに囲まれた状態」が常態 → **不確実性に耐える筋力** の土台作りが今やってること
- 「聞く / 考える」の判断軸を持つと、学習が **計画的** になる。どこで時間を使うかを自分で決められる = 学習の主導権が自分側に戻る

## 2026-04-21 Loop 6 まとめ — 関数

### 書いたコード

`examples/07_stats.go` で、点数のスライスから統計情報を返す2つの関数を **自分で** 定義した。

- `minMax(nums []int) (int, int)` — スライスの中から最大値と最小値を返す
- `sumAvg(nums []int) (int, float64)` — 合計と平均を返す

Loop 5 までは **「既にある関数を使う」側** だったが、今回はついに **関数を自分で書く側** に回った。袋（スライス）を受け取って、中をぐるっと見て、結果を2つ同時に返す—この "部品" を自分の手で作ったのが今日のハイライト。

### 学んだ3つの概念

1. **関数シグネチャの5パーツ構造**
   - `func 名前(引数 型) (戻り値の型)` の全体像
   - 引数 `(nums []int)` = 「**int のスライス** を **nums という名前** で受け取る」
   - 戻り値 `(int, int)` = 「**int を2つ返す** という契約」
   - **複数戻り値の時だけ括弧が必要**（1つなら `) string` とだけ書く）
   - シグネチャは **関数の "外見から見える契約書"**。中身を読まなくても何が入って何が出るかがわかる

2. **複数戻り値の仕組み**
   - `return max, min` のように **カンマ区切り** で2つ返す
   - 受け取り側は `max, min := minMax(scores)` で **同時に** 受け取れる
   - **順番が超重要**: `(int, int)` の1番目と `return` の1番目が対応している
   - 順序を間違えても **コンパイルは通る** → 結果が逆になる（気づきにくいバグ）
   - **自販機の例え**: 2本同時に出る自販機。ラベル（型）と取り出し口（return）の両方で「1番目はコーラ、2番目は水」と約束してる

3. **スコープとシャドーイングの罠**
   - `:=` は「**箱を新しく作る**」、`=` は「**中身を入れ替える**」
   - `for` の中で `:=` を使うと、**内側に新しい変数が生まれる**（外側と同名でも別もの）
   - 外で用意した `max` を for の中で更新したいなら `=` を使わないといけない
   - **家と部屋のメタファー**: 部屋の中から家の家具は見えるし書き換えられる。でも部屋の中で作った家具は、部屋を出た瞬間消える

### 3箇所連続バグを全部自力で直した話

書き上げた直後、出力がおかしかった。コンパイルは通ってるのに結果が変。順番に仮説を立てて潰した。

| 修正前 | 修正後 | 何が直ったか |
|---|---|---|
| for の中で `max := score` と書いてた | その行を削除して外の `max` を使う | シャドーイング解消（家の `max` をちゃんと更新できるようになった） |
| `if max > score { max = score }` | `if max < score { max = score }` | 不等号の向きを逆に（**暫定王者より大きい時だけ** 更新するのが正解） |
| `if min < score { min = score }` | `if min > score { min = score }` | 同上（min は逆向き） |

### ここが学びの核

- シャドーイングは **コンパイルエラーにならない**。Goが「これシャドーイングしてますよ」とは警告してくれない → 気づくのは出力を見た時だけ
- 不等号の向きのバグも **コンパイルは通る** → 期待値と出力を比べて初めて気づく
- これが **ロジックバグ**（構文ミスじゃないエラー）の正体。**エラーメッセージに頼れない**
- でも **出力を見て → 仮説を立てて → 修正して → 再実行** のループを **自力で回せた**。これが今日一番デカい収穫

### エラーメッセージの読み方で覚えたこと

エラーの **冒頭** が「種類」を教えてくれる。ここだけ覚えるとエラーの地図が頭にできる。

| 冒頭 | 意味 |
|---|---|
| `syntax error:` | 書き方の文法ルール違反 |
| `undefined:` | 名前がどこにも定義されてない |
| `declared and not used` | 宣言したのに使ってない |
| `cannot use X as Y` | 型が合わない |

`un-` 接頭辞（否定）のパターン: `unused` / `undefined` / `undeclared` / `unexpected` — **分解すると読める**。英語の接頭辞を構造として掴むと、エラーが怖くなくなる。

今日詰まった最初の syntax error は **`If`（大文字）で書いてた** のが原因。Goの予約語は `if`（小文字）のみ。**Goは大文字/小文字を厳密に区別する** という Loop 1 の教訓がここで再発動した。

### メタ学習: 「聞く / 考える」プロトコルを実運用できた

昨日整理した仮説ベース質問プロトコルを、今日のループで実際に回した。

- 「**nums[0] はゼロ値ですか!**」 → 仮説ベースのベストフォーム。Yes/No で返せる粒度で聞けた
- 「**これって max 定義されてないエラー?**」 → 仮説ベースだったが **外してた** → でも間違いを通じて `undefined` と `syntax error` の違いを覚えた。**外しても学びが残るのが仮説ベースの強み**
- 5分ルール / 15分ルールを意識して粘り度を調整できた

### 裏ゴールへの繋がり

- **自分で関数を書けた** = プログラムの「部品化」を初めて自分の手でやった。ここから先はこの部品の組み合わせ
- **ロジックバグを自力で直した** = 「**自分で書ける・自分で直せる**」の明確な第一歩
- vibeコーディング期には絶対に得られない「**なぜ動かないかを推論する体験**」を積めた。コンパイルエラーは Goが教えてくれるが、ロジックバグは **自分の頭でしか直せない**。ここが筋トレの本丸

### 実感したメリット

先のプロトコル（2026-04-21 の「聞く/考える」判断プロトコル）で整理した「仮説ベース質問」の **実戦版の効果** を、今日はっきり身体で感じた。

| メリット | 中身 |
|---|---|
| **情報圧縮** | 「全部見て」じゃなく「この1行が怪しい」でピンポイントに渡せる。自分も相手も思考コストが軽い |
| **答え合わせモード** | 答えが返った瞬間に「当たり/外れ」が確定する。自分の理解の地図に印がつく |
| **外した時も学びになる** | 「`type` 側が怪しい」と言って外れたら、**なぜ `main` 側だったか** の理由が逆算で理解できる。外し方そのものが教材になる |

### 使えたテンプレ

今日のやり取りから逆算すると、**自分が使いやすい型** はこれだとわかった。

```text
「〇〇だと思うけど、XXXで合ってる?」
「原因は △△ 側じゃなく ◇◇ 側な気がする。合ってる?」
「ここまではわかった。残りは □□ の書き方だけ怪しい」
```

共通点:

- **自分の仮説がある** — Yes/No で返せる粒度
- **範囲が絞れてる** — 「全部」じゃなく「ここ」と言えてる
- **何までわかって何がわからないか** が質問文に埋め込まれてる

逆に昔のクセで出がちな NG形:

- 「動きません」 → 情報ゼロ。仮説もゼロ
- 「どう書けばいい?」 → 丸投げ。思考が止まってる合図

### 裏ゴールとの繋がり

「自分で書ける・自分で直せる」になるには、**仮説を立てる筋力** が前提だった。

- コードを書く時 → 「この型ならこう動くはず」の仮説
- エラーを読む時 → 「このメッセージならここが原因のはず」の仮説
- 質問する時 → 「ここが怪しいと思う」の仮説

全部 **仮説 → 検証** の同じ運動だと気づいた。質問の仮説ベース化は、実は **コードを書く/直す力そのもの** を鍛えてる。つまり「聞き方を良くする」は「聞かない力を育てる」に繋がる。遠回りに見えて直結してる。
