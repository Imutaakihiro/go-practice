# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## このリポジトリの目的

オライリー『実用Go言語』を読みながらGoを学ぶ**個人学習リポジトリ**。プロダクションコードではない。

ユーザーは **プログラミング自体がほぼ初めての完全な初学者**。Goが本格的に学ぶ初の言語であり、Python・JavaScript等の他言語の経験もほぼない。これまではAI支援中心のいわゆる "vibeコーディング" で、エラーを自力で直す経験が浅い。**「自分で書ける・自分で直せる」状態になること** が裏のゴール。

そのため、文法習得自体ではなく **エンジニアの基礎となる考え方** を身につけ、技術を **「自分ごと」として学ぶ姿勢** を育てることが本当のゴール。

## 学習ログの運用

`learning/<topic>/` をユニットに、トピック単位で学びを残す。各トピック内のファイルは役割で分けて単一責任を保つ:

- `learning/<topic>/plan.md` — 章を概念単位で分割したループ計画。進捗チェックボックスで管理
- `learning/<topic>/notes/XX_xxx.md` — 本の内容を**自分の言葉でまとめたノート**（ファインマン・テクニック）。本の節ごとに1ファイル
- `learning/<topic>/scrap.md` — Zennスクラップ風の学習ログ。`## YYYY-MM-DD HH:MM 見出し` で時系列に追記し、疑問→調査→解決の流れを残す
- `learning/<topic>/examples/` — そのトピックで書いた `.go` コード
- `learning/README.md` — トピック一覧表。新トピックを切ったらここに1行追加

各ファイルの違い:

| ファイル | 内容 | いつ書く |
|---|---|---|
| `plan.md` | 静的なロードマップ | トピック開始時に1回 |
| `notes/XX.md` | 整理された知識（教科書的） | 「読む」ステップで本を閉じた後 |
| `scrap.md` | 気づき・疑問・はまった話の時系列 | ループ中いつでも |
| `examples/` | 実際に動かしたGoコード | 「書く」ステップで |

新しい話題を始めるときの判断: 既存トピックに収まるなら追記、独立した概念なら新ディレクトリ。

## Go環境

- Go 1.26.2 (darwin/arm64)
- モジュールルート: リポジトリ直下の `go.mod` 1つ（`github.com/androots/go-practice`）
- 全 `examples/` 配下のコードはこの単一モジュールに属する

## よく使うコマンド

```bash
go run ./learning/<topic>/examples/<file>.go   # 単発実行（学習中の試し打ち）
go test ./learning/<topic>/...                 # パッケージ単位のテスト
gofmt -w .                                     # 整形（Goでは "正しい整形" は1つだけ）
go vet ./...                                   # 静的チェック（疑わしいコードを検出）
go mod tidy                                    # 依存関係を整理
```

## 教えるときの方針（重要）

ユーザーは "プログラミング自体ほぼ未経験" + "エラーを読み解く力を育てている途中"。実装/解説するときは:

- **プログラミングの基礎概念から説明**: 「変数」「関数」「型」「パッケージ」「コンパイル」「ターミナル」なども初出として扱い、毎回1行で噛み砕く。前提知識を当てにしない
- **他言語との比較は使わない**: 「Pythonでいうと」「JSなら」は伝わらない。比喩を使うなら **日常生活のもの**（料理・住所・書類・郵便など）
- **暗記より仕組み**: 「なぜそう書くのか」「なぜこの記法なのか」を必ず添える
- **1つの解説に概念を3つ以上詰めない**: 詰めすぎたら章を分けて段階的に提示する。ユーザーが「わかった」と言うまで次の概念に進まない
- **何が起きているかを先に、なぜは後追い**: コードの **見た目で何が起きるか** を見せてから、概念の名前を与える順序
- **エラーは一緒に読み解く**: `file:line:col: message` の構造から、英語の決まり文句の意味まで噛み砕く
- **抽象化を急がない**: インターフェイス・ジェネリクス・並行処理は必要になった時点で導入。最初から出さない
- **標準ライブラリ優先**: サードパーティ製ライブラリやフレームワークを安易に勧めない（まず素朴に書ける感覚を養う）
- 学んだ内容は対応する `notes/` と `scrap.md` に追記する流れで進める

## アーキテクチャ

学習用なので大袈裟な構造はない。`learning/<topic>/` は相互に独立しており、各トピックは1ループ約1.5時間の **4ステップ学習法** で進める:

1. **読む**（〜30分） — 本の該当節を読む
2. **書く**（〜30分） — `examples/` に自分の手でコードを書く
3. **読む（実コード）**（〜15分） — polaris などの実プロジェクトで同じ概念を探して読む
4. **まとめる**（〜15分） — `notes/` に整理した理解を、`scrap.md` に気づきを時系列で残す

各ループの開始時には、その節で扱う概念を全部入れた**具体的な課題**（ファイル名・お題・概念チェックリスト・完成イメージ）を提示する。
